# MASTER_LP_TEMPLATE 運用ルール（最重要）

このフォルダは「LP制作の家（固定の土台）」です。
目的は **速度・品質・保守性・再現性・資産化** を同時に満たすこと。
“その場しのぎ”で散らかした瞬間に、事業として崩壊します。

---

## 1. 3レイヤーの考え方（絶対ルール）

### A) 固定レイヤー（AI禁止・基本いじらない）
このテンプレの屋台骨。ここが壊れると全案件が死ぬ。
- ファイル構造（このREADMEで定義）
- 共通CSS設計（tokens.css / layout.css / reset.css）
- フォーム導線（index → thanks など）
- GA4計測、GAS連携（ある場合）
- 共通のヘッダー/フッター、基本のSEO/OGP

※ 固定レイヤーは「案件ごとに別実装」しない。

### B) 差し替えレイヤー（案件ごとに変える場所）
案件に合わせて変えていい領域。
- セクションの中身（文言・構成）
- セクションの見た目（装飾・レイアウト）
- 追加セクション（例：収入シミュレーション）
- 必要最小限のアニメーション（範囲指定して追加）

### C) 回収棚（資産化：当たり部品の保管場所）
案件や検証で「使える」と確定した部品は、必ず棚へ回収する。
- 置き場所：`components/variants/<component-name>/`
- 1部品＝1フォルダで完結（後述）

---

## 2. ディレクトリ構造（これを維持）

- `index.html`：メインLP
- `thanks.html`：送信完了ページ
- `privacy.html`：プライバシーポリシー
- `reset.css`：リセット
- `tokens.css`：色/余白/フォント等の設計トークン
- `layout.css`：レイアウト/共通ユーティリティ
- `script.js`：共通JS（ここを肥大化させない）
- `images/`：画像

### 部品エリア
- `components/`
  - `variants/`：オプション部品（回収棚）
    - `<component-name>/`
      - `<component-name>.html`
      - `<component-name>.css`
      - `<component-name>.js`
      - `README.md`

---

## 3. components と variants の意味

### `components/`
「部品置き場」の親フォルダ（カテゴリ）

### `components/variants/`
**案件によって使う/使わない“オプション部品”の棚**
- 共通で常時使うものと混ぜないための境界線
- ここに入っている部品は「棚から取り出して使う」前提

---

## 4. 回収棚（variants）の登録ルール（超重要）

### 4点セット必須
variantsに入れる部品は必ずこの4点セットで登録する。
1. `<component-name>.html`（セクションHTMLだけ）
2. `<component-name>.css`（その部品専用CSSだけ）
3. `<component-name>.js`（その部品専用JSだけ）
4. `README.md`（使い方・依存・注意点）

### 禁止
- ルート直下に部品ファイルを増殖させる
- htmlはここ、cssは別…みたいに分散配置
- READMEなしで棚に入れる（＝再利用不能）

---

## 5. 部品の使い方（取り出し手順）

1) `components/variants/<component-name>/` を開く  
2) READMEを読む（依存と使い方を確認）  
3) HTMLを `index.html` に貼る（推奨位置に）  
4) CSS/JSを読み込む（必要なら）
5) 動作確認（最低限）
   - スマホ実機で崩れない
   - フォーム導線が壊れてない
   - 計測が必要ならGA4 DebugViewで確認

---

## 6. やってはいけないこと（即メンテ地獄）

- 「ページ全体をいい感じに生成して」依頼に戻る
- 固定レイヤー（構造/命名/計測/導線）を案件ごとに変える
- 動きを全要素にばら撒く（重い・胡散臭い・壊れる）
- テストせず棚に回収する
- 当たり部品を棚に戻さず、案件フォルダに置きっぱなしにする

---

## 7. 更新ルール（資産化の回し方）

- 案件で当たりが出たら棚へ回収（variants化）
- 棚の部品を改善したら、READMEも必ず更新する
- “棚が増えるほど制作が速くなる”状態が正解


# 収入シミュレーション (Sim-Income)

応募者の不安を軽減するための、簡易収入シミュレーション部品です。
日給・週出勤日数・交通費をスライダーで操作し、月収・年収の目安を即座に表示します。

## 概要

- **目的**: 応募前の収入イメージ具体化・離脱防止
- **推奨設置位置**: 「待遇・福利厚生」セクションの直後、「応募CTA（フォーム）」の直前
- **依存関係**: なし（Vanilla JS / CSS）

## 導入手順

  └ components/
      └ variants/
          └ sim-income/
              ├ sim-income.html
              ├ sim-income.css
              └ sim-income.js
